# wasm-tinygo

`wasm-tinygo` is a browser bootstrap project for a future TinyGo-in-WASM toolchain.

Compatibility details live in [COMPATIBILITY.md](/home/seorii/dev/hancomac/wasm-tinygo/COMPATIBILITY.md).

This repository does **not** run the real TinyGo CLI in the browser yet. It now implements the first browser-side build-driver slice:

1. reuse the [emception](https://github.com/jprendes/emception) browser toolchain from a local app
2. run a Go/WASI build driver that writes a request/result contract
3. execute the returned tool plan with emception in the same browser session

## What is working

- `npm run dev` fetches and patches the published emception worker into `public/vendor/emception/emception.worker.js`
- `npm run dev` also builds `cmd/go-probe` into `public/tools/go-probe.wasm`
- the Go/WASI driver reads `/workspace/tinygo-request.json` and writes `/workspace/tinygo-result.json`
- the driver validates that the requested entry file is `package main` and defines `func main()`
- the default planner mode is now `tinygo`, with TinyGo-style `scheduler` / `panic` option validation
- the planner resolves TinyGo-style target profiles for `wasm` and `wasip1` from shared bootstrap target JSON, and emits target-aware `clang` / `wasm-ld` flags
- the driver scans same-directory `.go` files as one package, ignores `_test.go`, hidden, and `_`-prefixed Go files, honors target-matching `//go:build` constraints, and filters `*_js.go` / `*_wasip1.go` / `*_wasm.go` suffixes like a Go package loader
- when the entry file or an imported local package exists but all its files are filtered out by the current target/build tags, the driver now returns a specific exclusion diagnostic instead of a generic `not found` error
- the driver classifies stdlib-like imports into metadata and rejects relative or unresolved external module imports before emitting a build plan
- the driver reads the nearest `go.mod`, records the current module path in metadata, and recursively loads current-module package files for imports that stay inside that module prefix

## Why this shape

Directly compiling the upstream TinyGo CLI to browser WASM is blocked right away:

- the stock CLI imports serial/monitor codepaths that do not build cleanly for `wasip1/js`
- the TinyGo compiler pipeline expects process spawning for `clang`, linkers, and post-processing tools

emception already solves the second half for LLVM/Emscripten-style tools. This repo now adds a browser-side Go/WASI driver that can generate a structured build plan before a future trimmed TinyGo front-end is swapped in.

## Commands

```bash
npm install
npm run dev
```

Useful scripts:

- `npm run prepare:assets` downloads the emception worker and rebuilds the Go/WASI probe
- `npm run build` runs the asset preparation step and creates a production build
- `npm run check` runs TypeScript checking only
- `npm run test:host` verifies browser-host file materialization for nested planner output paths and emception worker patching for the mkdir bridge
- `npm run test:browser` runs a headless Chromium smoke test against the built app, booting emception and driving the TinyGo bootstrap flow through browser test hooks; in sandboxes that forbid loopback listen or browser launch it exits early instead of failing the whole suite
- `go test ./...` verifies the request/result driver contract
- `npm run test:wasi` runs the built WASI driver contract tests, including constrained multi-file package loading
